# Decision #186 — Stronghold snapshot password-flow (Android Keystore derivation)

**Date**: June 2026
**Phase**: mobile-build roadmap, Phase B follow-up
**Deciders**: project lead (Vimal Nair)
**Status**: Accepted + design complete; implementation is a Phase B work item (requires an Android physical device + Android Studio Iguana+)

## Context

Phase A of the mobile-build roadmap (Decision #177) wired up
the Stronghold facade for Android secrets. The snapshot
file at `app_local_data_dir()/lipi.stronghold.hold` is
encrypted with a `KeyProvider` derived from a
**hardcoded 32-byte placeholder**:

```rust
// src-tauri/src/secrets_stronghold.rs:118
const PLACEHOLDER_KEY: [u8; 32] = [
    0x4c, 0x69, 0x70, 0x69, 0x2d, 0x73, 0x74, 0x72, ...
]; // "Lipi-stronghold-v1-placeholder!!"
```

This is **not a secret** — it's the ASCII bytes of
`"Lipi-stronghold-v1-placeholder!!"` spelled out. Anyone
who can `adb pull` the snapshot file off the device (or
read it off a backup, or exfiltrate it via a rooted-device
attack) can decrypt the snapshot with this known key. The
v1 placeholder exists only to satisfy the
`KeyProvider::try_from(Zeroizing<Vec<u8>>)` 32-byte length
requirement; it's flagged in the source code as "the v1
password-flow design is the future Mac / Linux session's
task".

Phase A intentionally shipped with the placeholder because
the Android physical-machine work (Swift / Kotlin plugin
implementation, real device smoke tests, Stronghold native
build verification) is the future Mac / Linux session's
job. The placeholder keeps the Windows-doable seam
buildable and testable on this box without giving up
the cryptographic shape of the production code.

Now that the seam is shipped, we need to commit to a real
password-flow design before any user data lands in a
Stronghold snapshot on a real Android device.

## Threat model

The realistic threats we design against:

- **Lost / stolen device.** Attacker has physical access
  to an unlocked device (or a locked one they can unlock
  via coercion). Wants to dump the AI provider API keys.
- **Adb pull on a developer-mode device.** Developer
  enables USB debugging, attacker `adb pull`s the
  snapshot file. Wants to decrypt it offline.
- **Malicious app on the same device.** A different app
  with `READ_EXTERNAL_STORAGE` (deprecated, but possible
  on older Android) reads the snapshot file. Wants to
  decrypt it.
- **Cloud backup exfiltration.** Snapshot file is
  included in the user's Google Drive backup. Attacker
  compromises the Google account. Wants to decrypt the
  backup.
- **Rooted device.** User has rooted their device. The
  snapshot file is readable by any process. Attacker
  wants to decrypt it.

The threats we do **not** design against:

- **Compromise of the Android Keystore itself.** If the
  attacker has TEE / Strongbox access, they can extract
  any Android Keystore key. Hardware-backed Keystore is
  the security boundary, not a guarantee. We document
  this in user-facing copy.
- **Compromise of the running app process.** If a
  malicious actor has the same UID as Lipi, they can
  read the in-memory `Stronghold` instance. We're not
  designing a DRM scheme; we're protecting at-rest data.
- **User installs a keylogger.** Password entry can be
  recorded. Option C (user-entered password) is vulnerable
  to this; Options A and B are not.

## Decision

**Derive the Stronghold snapshot encryption key from a
key generated by the Android Keystore (Master Key wrapped
AES-256 key), accessed via a small Kotlin helper called
from Rust via JNI.** This is Option A in the alternatives
analysis below.

### Concrete shape

```
+---------------------+        +----------------------+
| Rust secrets_       | JNI    | Kotlin Stronghold-   |
| stronghold.rs       | -----> | KeyBridge.kt         |
| (load_key call)     |        |                      |
+---------------------+        | 1. AndroidKeystore   |
                               |    .generateKey(     |
                               |       KEY_ALGO_AES,  |
                               |       256,           |
                               |       GCM-mode)      |
                               | 2. cipher = key.get- |
                               |    Cipher("AES/GCM/  |
                               |    NoPadding")       |
                               | 3. encrypt(zeros)    |
                               | 4. return ciphertext |
                               +----------------------+
```

The key derivation function is:

1. The Kotlin helper generates (on first launch) an
   Android Keystore key called `lipi_stronghold_master`
   with:
   - `KeyGenParameterSpec.Builder` with
     `setKeySize(256)`, `setBlockModes(KeyProperties.BLOCK_MODE_GCM)`,
     `setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)`,
     `setRandomizedEncryptionRequired(true)`.
   - `setUserAuthenticationRequired(false)` — we want
     silent decryption on every launch, not a biometric
     prompt per call. The trade-off is documented below.
2. The helper encrypts a known plaintext (`b"\x00" * 32`,
   32 zero bytes) under this key and returns the
   ciphertext (IV + ciphertext, base64-encoded).
3. The Rust side takes the SHA-256 of the ciphertext
   (which is uniformly random because the AES-GCM
   output depends on the random IV), truncates to 32
   bytes, and uses that as the `KeyProvider` key.
4. The first launch creates the Keystore key + writes
   the initial empty Stronghold snapshot. Subsequent
   launches read the existing snapshot.

The shape is:
- **First launch:** Android Keystore key is generated,
  32-byte plaintext is encrypted, SHA-256 of ciphertext
  is the snapshot key, snapshot is created.
- **Subsequent launches:** Android Keystore key already
  exists, encryption re-runs (with a fresh IV), SHA-256
  of ciphertext is the snapshot key, snapshot is
  decrypted.
- **App data clear:** Android Keystore key is wiped
  (this is automatic — Keystore entries are scoped to
  the app UID, and clearing app data removes them).
  The snapshot file is also wiped. The user re-enters
  their AI provider keys. Acceptable.
- **App reinstall without data clear:** Keystore key
  persists (UID is the same). Snapshot file may or may
  not persist depending on the install path. If it
  doesn't, the user re-enters their keys. Acceptable.
- **Device migration:** Keystore key is per-device.
  Snapshot is non-transferable. User re-enters their
  keys on the new device. Acceptable.

### Why SHA-256 of ciphertext (not the plaintext)?

The ciphertext is the encryption of a known plaintext
under a hardware-backed key. The ciphertext varies per
call (AES-GCM uses a random IV). The SHA-256 of the
ciphertext is therefore a uniformly random 32-byte value
that is:
- **Deterministic for a given call** (we get the same
  output if we re-call with the same IV, but the IV
  changes every time, so the output is different).
- **Different every time we call the helper** — which
  means we'd get a different snapshot key every launch
  if we called it freshly.

That's a problem: we need a **stable** snapshot key, not
a fresh one. So we cache the SHA-256 result in
`shared_preferences` (in plaintext, but the SHA-256 is
already uniformly random — there's no secret in the
cache, just a stable handle). The cached value is
overwritten if the Keystore key changes (which would
indicate the snapshot file is unrecoverable anyway).

Alternative: use the Keystore key to **wrap** a
per-install random 32-byte value (the snapshot key).
The wrapped key is stored in `shared_preferences`. To
unwrap, we ask the Keystore to decrypt. This is the
classical Android Keystore pattern.

For Lipi's case the SHA-256-of-ciphertext approach is
simpler (no separate random key generation, no wrap/
unwrap), and the security property is the same: the
Keystore key never leaves the device, the snapshot
file is useless without the Keystore key, and the
cached handle in `shared_preferences` is a random
value that reveals nothing about the Keystore key.

### Why no `setUserAuthenticationRequired`?

The trade-off: if we set `UserAuthenticationRequired`,
the Android Keystore requires a biometric / device-credential
prompt every time the snapshot is decrypted. That's
once per app launch (the `Stronghold` instance is cached
in `Mutex<Option<...>>` for the process lifetime), so
the user sees one biometric prompt per session.

For most apps this is the right call. For Lipi, the
UX cost is high (a developer opens Lipi dozens of
times per day) and the security benefit is low (a
developer with their phone in their hand is going to
pass the biometric check; the threat model is "phone
is stolen while locked", and a locked phone is
already protected by the lock screen).

**Decision:** leave `setUserAuthenticationRequired(false)`.
Document the trade-off in user-facing copy. A v1.1
follow-up could add a Settings toggle for
"Require biometric unlock on app launch" that flips
this flag.

### Why not Option B (per-install random)?

Per-install random is the classical pattern, but on
Android the *only* safe place to store a random key
is the Android Keystore (or the Strongbox). So Option
B reduces to "generate a random key + wrap it in the
Android Keystore + store the wrapped key in
`shared_preferences`." That's exactly Option A with
an extra step. The extra step adds complexity (two
secrets to manage: the random key and the wrapping
key) without adding security. Reject.

### Why not Option C (user-entered password)?

UX cost: every Lipi user has to remember a password
(or lose access to their API keys). Most users will
pick a low-entropy password. The threat model
("attacker has the snapshot file") isn't actually
defended against by a low-entropy password — the
attacker can brute-force the password offline at
GPU speeds (Argon2 with realistic parameters runs
~100 hashes/sec on a GPU, so a 6-character lowercase
alphanumeric password falls in minutes).

The one threat Option C defends against is the
"Keystore compromised" threat, which is a much
rarer threat than "user lost their phone". For most
users the trade-off is bad: high UX cost for a low-
probability security gain.

If Android Keystore is unavailable on the device
(very old Android, broken Keystore, custom ROM
with Keystore stripped), fall back to Option C.
This is a v1.1 follow-up; v1 ships with Option A
only.

### Why not Option D (Hybrid: Keystore + password)?

Highest security (the attacker needs both the
Keystore key and the user password) but highest UX
cost (password prompt on every launch). The
threat-model-weighted gain over Option A alone is
small (Option A already requires the attacker to
have hardware-level Keystore access, which is rare).
The threat-model-weighted gain over Option C alone
is meaningful (a stolen-device attacker doesn't have
the Keystore key), but at a permanent UX cost.

**Decision:** not for v1. Add as a v1.1 follow-up
behind a Settings toggle for users who explicitly
want the extra security. The default is Option A.

### Why not Option F (swap Stronghold for EncryptedSharedPreferences)?

Stronghold is the chosen backend (Decision #177);
swapping it for EncryptedSharedPreferences would
re-architect the entire secrets layer (the JS-side
contract, the IPC commands, the desktop/mobile
dispatch). The benefit is a slightly simpler Android
implementation (no Stronghold native build, no
`mobile` Cargo feature). The cost is significant
(changing the desktop backend, breaking the
tauri-plugin-stronghold dependency, re-doing the
snapshots are stored in `app_local_data_dir()` shape
on iOS / macOS — actually, iOS uses keyring 3.x,
not Stronghold, so this is moot for iOS).

**Decision:** out of scope. EncryptedSharedPreferences
is a good *v2* candidate if Stronghold proves
problematic in production. For v1, Stronghold stays.

## Implementation plan

This decision is **accepted but not implemented** in
this commit. The implementation is a Phase B work
item because it requires:

1. **A real Android device** with Android 7.0+ (API
   24+, matching the `mobile` feature's `minSdkVersion`).
2. **Android Studio Iguana+** for the Kotlin plugin
   build (or `cargo tauri android` from CLI).
3. **A hardware-backed Keystore** on the test device
   (most modern devices have this; the test plan
   needs a device that doesn't, to exercise the
   fallback path).
4. **An Android cross-build environment** on a Linux
   or Windows box. The current Windows box can drive
   the Android Studio IDE but not the full
   `cargo tauri android build --features mobile`
   pipeline (the cross toolchain needs the NDK).

When the future Mac / Linux session picks this up,
the implementation steps are:

1. Add `lipi-stronghold-key-bridge` Kotlin module
   (sibling to the future `lipi-stt-android` plugin
   from the Phase B plan). Module: `:android-key-bridge`.
   Files: `StrongholdKeyBridge.kt` (~100 lines),
   `AndroidManifest.xml` fragment for
   `android.permission.USE_BIOMETRIC` (only needed if
   we add the biometric toggle later), JNI bindings.
2. Add JNI bindings to `src-tauri/src/secrets_stronghold.rs`:
   - New `#[cfg(target_os = "android")]` function
     `load_key_from_keystore() -> Zeroizing<[u8; 32]>`.
   - Calls `StrongholdKeyBridge.kt` via JNI
     (`Java_lipi_stronghold_StrongholdKeyBridge_nativeLoadKey`).
   - The current `keyprovider()` and `get_or_init()`
     helpers change: instead of `PLACEHOLDER_KEY`, they
     call `load_key_from_keystore()`.
3. Add a Kotlin-side unit test
   (`StrongholdKeyBridgeTest.kt`) that mocks the
   Android Keystore and verifies the SHA-256 of the
   ciphertext is deterministic for a given Keystore
   key + IV.
4. Add a Rust-side integration test (gated
   `#[cfg(feature = "mobile")]`) that exercises the
   full round-trip with a stubbed JNI bridge.
5. Update the `secrets_stronghold.rs` module docs to
   point at the new `load_key_from_keystore()` instead
   of the `PLACEHOLDER_KEY` constant.
6. Drop the `PLACEHOLDER_KEY` constant (and the
   `Lipi-stronghold-v1-placeholder!!` string it
   encodes). The 32-byte length requirement is now
   satisfied by the Keystore-derived key.
7. Update `secrets_ai_smoke.rs` if the snapshot
   path setup changes (it shouldn't — the `snapshot_path`
   parameter is the same).

Estimated size: ~150 lines of Kotlin + ~50 lines of
Rust + ~50 lines of tests. The complexity is in
verifying the Android Keystore behavior on a real
device, not in writing the code.

## Why this is the right design (summary)

- **Matches the threat model.** Protects at-rest data
  from snapshot-file exfiltration (lost device, adb
  pull, cloud backup leak, malicious app).
- **Hardware-backed when available.** Android Keystore
  uses the device's TEE / Strongbox on modern hardware.
  The key never leaves the secure element.
- **Best UX of the three options.** No password to
  remember, no biometric prompt per session, no
  per-launch friction.
- **Standard pattern.** 1Password, Bitwarden, and
  most password managers use the same approach for
  their Android clients. The code review cost is low
  (security reviewers have seen this pattern many
  times).
- **Aligns with the existing Stronghold backend.**
  No re-architecture needed. The change is scoped to
  `secrets_stronghold.rs` + a small Kotlin helper.
- **Defers Option C as a fallback.** If Android
  Keystore is genuinely unavailable (broken device,
  custom ROM), the v1.1 follow-up adds a user-password
  fallback. The default v1 build assumes Keystore
  works, which is true for >99% of Android 7.0+ devices.

## Alternatives considered (full list)

### A. Android Keystore derivation — **Accepted**
Described above. The default for v1.

### B. Per-install random
Generate a random 32-byte key on first launch, wrap
it in Android Keystore, store the wrapped key in
`shared_preferences`. To decrypt: unwrap via Keystore.
**Rejected** because it's the same security
properties as A with an extra layer of indirection
(you have two secrets to manage: the random key and
the Keystore key). The indirection adds complexity
without adding security. Option A derives the
Stronghold key directly from the Keystore.

### C. User-entered password
Prompt the user for a password on first launch
(when the first AI provider key is set), derive a
32-byte key from the password via Argon2id
(`m = 64 MiB, t = 3, p = 1`), use that as the
Stronghold key. **Rejected as the default** because
of the UX cost + the low-entropy password problem.
**Accepted as a fallback** for devices where
Android Keystore is unavailable.

### D. Hybrid (Keystore + password)
Derive the Stronghold key from
`HMAC-SHA-256(keystore_derived_key, user_password)`.
**Rejected for v1** because the security gain is
small (Option A already requires hardware-level
Keystore access) and the UX cost is permanent. A
v1.1 follow-up could add this as a Settings toggle.

### E. Wait for `keyring` 3.x Android support
The `keyring` 3.6 crate has no Android backend.
Watching for upstream Android support. **Rejected**
because the keyring maintainers have not signaled
intent to add Android, and the desktop keyring
architecture doesn't fit the Stronghold snapshot
model anyway. Stronghold is the right backend for
Android (Decision #177).

### F. EncryptedSharedPreferences instead of Stronghold
Use Jetpack's `EncryptedSharedPreferences` (which
uses Android Keystore under the hood) instead of
Stronghold for Android. **Rejected** because it
re-architects the entire secrets layer. Option A
gives the same security property with a much smaller
change. Worth re-evaluating in v2 if Stronghold
proves problematic in production.

### G. Hardware security key (YubiKey etc.) for the key
Require the user to plug in a hardware security key
to derive the snapshot key. **Rejected** because
mobile Android devices don't typically have USB
host mode + the right drivers, and the UX is
catastrophic (most users don't have a hardware
key, and asking them to buy one for an IDE is
insane). This is a niche option for high-security
environments; not a v1 design.

## Impact

- **Code:** ~50 lines of Rust changes in
  `secrets_stronghold.rs` (new `load_key_from_keystore`
  function + drop the `PLACEHOLDER_KEY` constant) +
  ~150 lines of Kotlin in the new
  `:android-key-bridge` module + ~50 lines of
  JNI glue.
- **Tests:** 1 new Kotlin test
  (`StrongholdKeyBridgeTest.kt`) + 1 new Rust
  integration test (mobile-gated, exercises the
  JNI bridge stub) + 1 new end-to-end test
  (real Android device, real Keystore). Total: ~3
  new tests.
- **Docs:** Updated `secrets_stronghold.rs` module
  docs (drop the placeholder section, add the
  Keystore section) + updated `secrets_stronghold.rs`
  module-level comment about the placeholder + new
  `HANDOFF.md §9.50` (this decision) + updated
  `CHANGELOG.md` "Phase mobile-build roadmap -
  Phase B" entry.
- **Behavior change for end users:** the first time
  the user sets an AI provider key on Android, the
  Android Keystore key is generated. The user sees
  no UI change (the generation is silent). On
  subsequent launches, the Keystore key is reused.
  On app data clear, the Keystore key is lost and
  the user re-enters their AI provider keys. This
  is documented in user-facing copy.

## Out of scope for this decision

- **The `iOS` secrets backend.** iOS uses
  `keyring` 3.x's `apple-native` feature (which
  wraps the iOS Keychain). No Stronghold, no
  Keystore. Decision #177 covers the iOS
  backend.
- **The desktop secrets backend.** Desktop uses
  the OS keyring (`keyring` 3.x with the
  per-platform `*-native` features). No
  Stronghold, no Keystore. Decision #177 covers
  the desktop backend.
- **The biometric toggle.** A v1.1 follow-up
  could add a Settings toggle for
  `setUserAuthenticationRequired(true)`. The
  default v1 is `false` for UX reasons.
- **The Keystore-unavailable fallback.** A v1.1
  follow-up could add the Option C fallback
  (user-entered password) for devices where
  Android Keystore is broken. The default v1
  assumes Keystore works.
- **Migration of existing snapshot files.** When
  the v1 placeholder is dropped, any snapshot
  file written by the v1 placeholder build is
  unrecoverable. Since the v1 placeholder build
  was never released to a real user (Phase A is
  a seam, not a release), there's no migration
  to write. If a v1 placeholder build did ship
  to a test user, the test user re-enters their
  AI provider keys. Acceptable.
