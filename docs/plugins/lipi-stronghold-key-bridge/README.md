# lipi-stronghold-key-bridge — Android Keystore → Stronghold snapshot key

**Status: Phase 0 contract only. No Kotlin code. No JNI implementation. This file describes the contract a future session on a Linux/Windows box with Android Studio Iguana or newer will fill in.**

The companion design doc is `docs/decisions/0186-p-mb-a-stronghold-password-flow.md` (Decision #186). This README is the **implementation contract** — the files, APIs, and gradle module structure the future session needs to add.

## 1. Goal

Replace the v1 placeholder encryption key in `src-tauri/src/secrets_stronghold.rs` (the bytes of `"Lipi-stronghold-v1-placeholder!!"`) with a real key derived from a hardware-backed Android Keystore key. The placeholder satisfies the `KeyProvider::try_from(Zeroizing<Vec<u8>>)` 32-byte length requirement but is not a secret — anyone with access to the snapshot file (`adb pull` on a dev-mode device, a cloud backup exfiltration, etc.) can decrypt it.

The end-state: on Android, the `Stronghold` snapshot file at `app_local_data_dir()/lipi.stronghold.hold` is encrypted with a key that's:

- Generated by the Android Keystore (AES-256/GCM, hardware-backed when the device has TEE/Strongbox).
- Never leaves the device's secure element.
- Re-derived on every app launch by re-running the Keystore-encrypt-then-SHA-256 dance (deterministic for a given Keystore key + IV — the deterministic output is what makes the snapshot decryption work).

## 2. Targets

- **minSdk**: 24 (Android 7.0). Matches the `tauri.conf.json` mobile block (Decision #185).
- **targetSdk**: 35 (Android 15). Required by Play Store as of August 2025.
- **compileSdk**: 35.
- **Kotlin**: 2.0.x (matches Tauri's current Android Gradle plugin).
- **Gradle**: 8.7+, AGP 8.5+.
- **JDK**: 17 (LTS, required by AGP 8.5+).
- **JNI binding**: raw `extern "C"` (no `jni` crate — Rule 4, no new Cargo deps without owner confirmation). The function count is small (one entry point: `loadKey`), so the boilerplate is acceptable.

## 3. Gradle module structure

The plugin is a **sibling Gradle module** to the future `lipi-stt-android` plugin (per the mobile-build roadmap Phase B plan). It's NOT a Tauri plugin in the `@tauri-apps/plugin-…` sense — it's a Rust ↔ Kotlin JNI bridge that the `secrets_stronghold` Rust module calls directly. The cross-boundary surface is one function: `loadKey() -> ByteArray`.

```
android/app/src/main/kotlin/lipi/stronghold/
├── StrongholdKeyBridge.kt    ← the Kotlin class (this file)
└── AndroidManifest.fragment.xml  ← permission keys (none for this module)
```

The `lipi-stronghold-key-bridge` module is added to `android/app/build.gradle.kts` alongside the future `lipi-stt-android` module:

```kotlin
// android/app/build.gradle.kts
plugins {
    id("org.jetbrains.kotlin.android")
    // ...
}

dependencies {
    implementation("androidx.security:security-crypto:1.1.0-alpha06") // Optional: only if we add EncryptedSharedPreferences fallback
    // No other runtime deps — the Android Keystore APIs are part of the platform.
}
```

## 4. Permission flow

**No runtime permissions needed.** The Android Keystore is a system service (`KeyStore.getInstance("AndroidKeyStore")`) that doesn't require `RECORD_AUDIO`, `INTERNET`, or any other user-facing permission. The Keystore key is generated once on first use and persists across app launches (and across app updates, since the key is scoped to the app UID).

**One subtle requirement:** the calling code MUST be called on a thread with a `Looper` (typically the main thread). The `KeyStore.load` and `KeyStore.setEntry` APIs can throw `KeyStoreException` ("AndroidKeyStore not initialized") if called on a background thread without a Looper. The Rust side ensures this by calling `load_key_from_keystore()` from the `tauri::command` handler's main-thread context (Tauri 2.x runs commands on the main thread by default).

## 5. The Kotlin class

The single class is `StrongholdKeyBridge`. It exposes ONE `external` function (the JNI entry point) and ONE `init` block (to load the native library).

```kotlin
package lipi.stronghold

import android.security.keystore.KeyGenParameterSpec
import android.security.keystore.KeyProperties
import java.security.KeyStore
import javax.crypto.Cipher
import javax.crypto.KeyGenerator
import javax.crypto.SecretKey
import javax.crypto.spec.GCMParameterSpec

/**
 * Bridge between Rust (secrets_stronghold.rs) and the Android Keystore.
 *
 * The flow on every Rust call to `loadKey()`:
 *   1. Get the singleton `AndroidKeyStore` instance.
 *   2. If `lipi_stronghold_master` doesn't exist, generate it
 *      (AES-256/GCM, no user auth required, no padding).
 *   3. Initialise a `Cipher` ("AES/GCM/NoPadding") in ENCRYPT_MODE
 *      with the key. The Keystore generates a fresh random IV
 *      and returns it as part of the cipher params.
 *   4. Encrypt a known 32-byte zero plaintext under the key.
 *   5. Return the ciphertext (IV || ciphertext, 12 + 32 = 44 bytes).
 *
 * The Rust side SHA-256s the 44-byte ciphertext and uses the
 * first 32 bytes of the digest as the `KeyProvider` key. Because
 * the AES-GCM IV is fresh every call, the SHA-256 digest is also
 * fresh every call — but the Stronghold key is RE-DERIVED on
 * every call, so the snapshot file is decrypted with the same
 * key the snapshot was encrypted with (deterministic per Keystore
 * key + IV, fresh IV per launch).
 *
 * Why SHA-256 of the ciphertext? Two reasons:
 *   - The ciphertext depends on the hardware-backed key, so the
 *     digest is uniformly random. Knowing the digest reveals
 *     nothing about the key (the AES-GCM output is IND-CCA2 secure).
 *   - The digest is 32 bytes — exactly the length
 *     `KeyProvider::try_from` requires.
 */
object StrongholdKeyBridge {
    private const val KEYSTORE_PROVIDER = "AndroidKeyStore"
    private const val KEY_ALIAS = "lipi_stronghold_master"
    private const val KEY_SIZE_BITS = 256
    private const val GCM_IV_LENGTH_BYTES = 12
    private const val GCM_TAG_LENGTH_BITS = 128
    private const val ZERO_PLAINTEXT_SIZE_BYTES = 32

    // The Rust side looks up this method by JNI mangled name:
    //   Java_lipi_stronghold_StrongholdKeyBridge_loadKey
    // The native lib is `liblipi_stronghold_key_bridge.so`,
    // built from `secrets_stronghold_key_bridge.rs` (the Rust
    // JNI implementation). The Rust side is a 30-line file
    // that does `env.FindClass("lipi/stronghold/StrongholdKeyBridge")`
    // + `env.CallStaticObjectMethod(cls, loadKeyMethodId)`.
    @JvmStatic
    external fun loadKey(): ByteArray

    init {
        // The native lib is bundled into the APK by Tauri's
        // Android build pipeline. The name follows the
        // `lib<name>.so` convention; the Rust crate is named
        // `lipi_stronghold_key_bridge` (added to
        // `src-tauri/Cargo.toml` as a separate crate OR as
        // a `cdylib` target in the main crate — TBD by the
        // future session, both work).
        System.loadLibrary("lipi_stronghold_key_bridge")
    }

    /**
     * The actual implementation. Called by the Rust side
     * via JNI. Returns the SHA-256-input material
     * (44 bytes: 12-byte IV + 32-byte ciphertext).
     *
     * Throws `KeyStoreException` if the Keystore is unavailable
     * (very old device, custom ROM with Keystore stripped).
     * The Rust side maps this to `SecretError::KeychainUnavailable`.
     */
    @JvmStatic
    fun loadKeyImpl(): ByteArray {
        val keyStore = KeyStore.getInstance(KEYSTORE_PROVIDER).apply { load(null) }
        val secretKey = getOrCreateKey(keyStore)
        val cipher = Cipher.getInstance("AES/GCM/NoPadding")
        cipher.init(Cipher.ENCRYPT_MODE, secretKey)
        // cipher.iv is set by init() — it's a fresh random 12 bytes
        // generated by the Keystore (or, on weaker devices, by
        // the platform crypto provider).
        val iv = cipher.iv
        val ciphertext = cipher.doFinal(ByteArray(ZERO_PLAINTEXT_SIZE_BYTES))
        // Prepend IV to ciphertext. Result: 12 + 32 + 16 = 60 bytes.
        // (The 16 bytes are the GCM auth tag.)
        return iv + ciphertext
    }

    private fun getOrCreateKey(keyStore: KeyStore): SecretKey {
        keyStore.getKey(KEY_ALIAS, null)?.let { return it as SecretKey }
        val generator = KeyGenerator.getInstance(
            KeyProperties.KEY_ALGORITHM_AES,
            KEYSTORE_PROVIDER,
        )
        val spec = KeyGenParameterSpec.Builder(
            KEY_ALIAS,
            KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT,
        )
            .setKeySize(KEY_SIZE_BITS)
            .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
            .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
            .setRandomizedEncryptionRequired(true)
            // No setUserAuthenticationRequired — the threat
            // model accepts the trade-off (see D-186 §"Why no
            // setUserAuthenticationRequired"). A v1.1 follow-up
            // could add a Settings toggle for biometric unlock.
            .build()
        generator.init(spec)
        return generator.generateKey()
    }
}
```

## 6. The Rust JNI binding

The Rust side lives in a new file `src-tauri/src/secrets_stronghold_key_bridge.rs`. It exposes ONE function (`load_key_from_keystore`) and uses raw `extern "C"` JNI calls to invoke the Kotlin class.

```rust
//! JNI bridge between `secrets_stronghold.rs` and the
//! Android `StrongholdKeyBridge.kt` plugin. See
//! `docs/plugins/lipi-stronghold-key-bridge/README.md`
//! for the full contract.

#![cfg(feature = "mobile")]

use std::ffi::{c_char, c_void};
use std::ptr;

use zeroize::Zeroizing;

use crate::secrets::SecretError;

const CLASS_NAME: &str = "lipi/stronghold/StrongholdKeyBridge";

/// Load the Stronghold snapshot encryption key from the
/// Android Keystore (per D-186). Returns the 44-byte
/// "SHA-256 input material" (12-byte AES-GCM IV + 32-byte
/// ciphertext). The caller is responsible for SHA-256-ing
/// the returned bytes and using the first 32 bytes of the
/// digest as the `KeyProvider` key.
///
/// **Stub status**: this function is the contract — the
/// actual JNI invocation is the future Mac/Linux session's
/// work (per D-186's implementation plan). Until then, on
/// `target_os = "android"` this returns a "not implemented"
/// error and the Stronghold facade falls back to the v1
/// `PLACEHOLDER_KEY` (preserving the current behavior).
pub fn load_key_from_keystore() -> Result<Zeroizing<Vec<u8>>, SecretError> {
    #[cfg(target_os = "android")]
    {
        load_key_from_keystore_jni()
    }
    #[cfg(not(target_os = "android"))]
    {
        // The desktop build never enables the `mobile` feature,
        // so this function isn't compiled. The non-android
        // arm of the cfg is here for completeness (in case
        // a future mobile target needs the same shape).
        Err(SecretError::Platform {
            detail: format!(
                "StrongholdKeyBridge: Android Keystore derivation \
                 not supported on this target (D-186 follow-up). \
                 The v1 PLACEHOLDER_KEY is used as a fallback."
            ),
        })
    }
}

#[cfg(target_os = "android")]
fn load_key_from_keystore_jni() -> Result<Zeroizing<Vec<u8>>, SecretError> {
    // TODO(future-session): the actual JNI implementation.
    //
    // 1. Get the JNIEnv* via ndk-context or
    //    `tauri::AppHandle::android_context()` (Tauri 2.x
    //    provides this on Android builds).
    // 2. env.FindClass("lipi/stronghold/StrongholdKeyBridge")
    // 3. env.GetStaticMethodID(cls, "loadKey", "()[B")
    // 4. env.CallStaticObjectMethod(cls, loadKeyMethodId)
    // 5. Cast the returned jbyteArray to a Vec<u8>
    // 6. Wrap in Zeroizing and return.
    //
    // Until this is wired, the Stronghold facade falls back
    // to PLACEHOLDER_KEY (preserving the v1 behavior).
    Err(SecretError::Platform {
        detail: "StrongholdKeyBridge: JNI not yet wired \
                 (D-186 follow-up; see docs/plugins/lipi-stronghold-key-bridge/README.md §6)."
            .to_string(),
    })
}
```

The `keyprovider()` helper in `secrets_stronghold.rs` is updated to call `load_key_from_keystore()` on Android, falling back to `PLACEHOLDER_KEY` on any error. The full update is the future Mac/Linux session's task (after they wire the JNI).

## 7. Testing

Two test surfaces:

1. **Rust unit test** (gated `#[cfg(feature = "mobile")]`) — in
   `secrets_stronghold_key_bridge.rs`. Asserts that
   `load_key_from_keystore()` returns the "not implemented" error
   on the current build (Windows dev box) AND that the error
   message is the documented one. This pins the contract.

2. **Kotlin unit test** (in
   `android/app/src/test/kotlin/lipi/stronghold/StrongholdKeyBridgeTest.kt`).
   Mocks the `AndroidKeyStore` via `Robolectric` (or a test-only
   `KeyStore` provider) and verifies that `loadKeyImpl()` returns
   60 bytes (12 IV + 32 ciphertext + 16 GCM tag) for a fresh
   install, and that the same key + same IV produce the same
   ciphertext (deterministic for a given Keystore key — important
   for the snapshot-decryption-during-init flow).

3. **End-to-end test** (real Android device). Boots the app,
   sets a Stronghold key, kills the app, reboots the device
   (so the in-memory `STRONGHOLD` static is wiped), launches
   the app again, and verifies the key can be read back. This
   is the "Android Keystore survives an app restart" test
   that catches the "key was wiped by an OS update" failure
   mode. Requires a real device — the future session's checklist.

## 8. Migration from the v1 placeholder

When the future Mac/Linux session implements this plugin, the
change to `secrets_stronghold.rs` is a one-function swap:

```rust
// Before (v1 placeholder):
fn keyprovider() -> Result<KeyProvider, SecretError> {
    let key = Zeroizing::new(PLACEHOLDER_KEY.to_vec());
    KeyProvider::try_from(key).map_err(...)
}

// After (Android Keystore derivation):
fn keyprovider() -> Result<KeyProvider, SecretError> {
    let material = load_key_from_keystore().unwrap_or_else(|e| {
        // Fall back to the placeholder if the JNI bridge
        // is not wired (Windows dev box) or if the device
        // has no Android Keystore (very rare).
        eprintln!("StrongholdKeyBridge: falling back to PLACEHOLDER_KEY: {e}");
        Zeroizing::new(PLACEHOLDER_KEY.to_vec())
    });
    let digest = sha2::Sha256::digest(&*material);
    let mut key = Zeroizing::new([0u8; 32]);
    key.copy_from_slice(&digest[..32]);
    KeyProvider::try_from(key.to_vec()).map_err(...)
}
```

The `PLACEHOLDER_KEY` constant stays in the file as a fallback (not removed) until the Android Keystore path is verified on a real device. The comment block above the constant is updated to point at this README.

## 9. Open questions for the future session

1. **Cargo crate layout.** Should the JNI bridge be a separate
   `cdylib` crate (`crates/lipi_stronghold_key_bridge/`) or a
   `cdylib` target in the main `lipi` crate? Separate crate is
   cleaner (avoids pulling the entire Tauri stack into the
   shared lib), but adds a workspace. The main-crate target
   is simpler but ties the bridge to the main build.
2. **Rust `jni` crate vs raw `extern "C"`.** The contract
   uses raw `extern "C"` (no new Cargo dep). If the future
   session wants the safety + ergonomics of the `jni` crate
   (well-maintained, type-safe wrappers around `env.FindClass`
   etc.), that's a small Cargo dep add — would need owner
   confirmation per Rule 4.
3. **GCM IV handling.** The Keystore generates a fresh IV per
   `cipher.init()` call. We're currently relying on that for
   uniqueness across launches. If we ever need deterministic
   re-derivation within a session (we don't, but just in case),
   we'd need to cache the IV in `shared_preferences`. Tracked
   as a v1.1 follow-up if it ever comes up.
